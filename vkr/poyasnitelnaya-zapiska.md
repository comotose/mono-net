## АННОТАЦИЯ

Пояснительная записка содержит постановку и программу решения задачи «Разработка веб-платформы социальной сети MONO».

Веб-приложение MONO разработано на языке программирования PHP 8.1+ с использованием фреймворка Laravel 10, системы управления базами данных MySQL, клиентских технологий Blade, Alpine.js, Tailwind CSS и Vite. Для обмена сообщениями в реальном времени применяются Laravel Echo, протокол Pusher и сервер Soketi. Программный продукт предназначен для работы в операционной системе семейства Windows/Linux под управлением веб-сервера Apache или Nginx и отлажен на данных контрольного примера.

Платформа MONO предназначена для организаций и учебных заведений, нуждающихся в компактной социальной среде: публикация контента, подписки на пользователей, обмен личными сообщениями, уведомления о событиях. Результаты работы могут быть использованы как основа корпоративной или образовательной социальной сети.

Взаимосвязь с другими задачами: продукт интегрируется с почтовым сервером (верификация email), файловым хранилищем сервера и внешним WebSocket-сервисом; REST API (Laravel Sanctum) обеспечивает возможность подключения мобильных клиентов.

## ВВЕДЕНИЕ

В современных условиях цифровизации образования и корпоративных коммуникаций возрастает потребность в специализированных веб-платформах, обеспечивающих обмен информацией между участниками сообщества без избыточного функционала крупных социальных сетей [1]. Массовые платформы не всегда соответствуют требованиям конфиденциальности учебных заведений и небольших организаций, что обусловливает актуальность разработки автономных решений [2].

**Актуальность темы** определяется необходимостью создания лаконичной социальной среды с поддержкой публикаций, комментариев, подписок, личных сообщений и уведомлений в реальном времени. Подобные системы применяются для организации внутреннего общения студентов, преподавателей и сотрудников, а также для формирования профессиональных сообществ [3].

**Цель проекта** — разработка веб-платформы социальной сети MONO, обеспечивающей регистрацию пользователей, ведение ленты публикаций, социальное взаимодействие и обмен сообщениями с доставкой в режиме реального времени.

**Задачи проекта:**

1. Изучить предметную область социальных веб-платформ и описать бизнес-процессы взаимодействия пользователей.
2. Спроектировать структуру базы данных и диаграмму вариантов использования системы.
3. Сформулировать требования к программному продукту и описать входную и выходную информацию.
4. Реализовать серверную и клиентскую части приложения на базе Laravel, MySQL и современных frontend-технологий.
5. Выполнить тестирование программного продукта и составить руководство пользователя.

**Теоретической основой** послужили работы по проектированию баз данных (Стружкин, Годин [4]), разработке веб-приложений (Полуэктова [5]), официальная документация Laravel [6] и MySQL [7], а также нормативные документы в области информационных технологий (ГОСТ 7.32-2001 [8], ГОСТ 34.602-2020 [9]).

**Область применения результатов** — учебные заведения, небольшие организации и проектные команды, которым требуется контролируемая социальная платформа с возможностью развёртывания на собственном сервере. Внедрение продукта позволяет сократить зависимость от внешних сервисов и обеспечить хранение данных на территории организации.

## 1 ПОСТАНОВКА ЗАДАЧИ

### 1.1 Описание предметной области

Предметной областью является функционирование социальной веб-платформы MONO — системы, предназначенной для регистрации пользователей, публикации контента, социального взаимодействия и обмена личными сообщениями.

На предприятии (учебном заведении) реализуются следующие **бизнес-процессы**:

1. **Регистрация и аутентификация** — создание учётной записи, вход в систему, восстановление пароля.
2. **Ведение профиля** — редактирование имени, email, биографии, аватара, тегов интересов, настроек уведомлений.
3. **Публикация контента** — создание постов с текстом и изображениями (до 9 файлов), удаление собственных публикаций.
4. **Взаимодействие с контентом** — постановка реакций (сердце, огонь, аплодисменты, глаза), комментирование, ответы на комментарии (ветвление).
5. **Социальный граф** — подписка и отписка от пользователей, просмотр списков подписчиков и подписок, поиск пользователей.
6. **Личные сообщения** — текстовые сообщения, файлы, изображения, голосовые записи; доставка в реальном времени через WebSocket.
7. **Уведомления** — оповещения о новых сообщениях, подписчиках, реакциях и комментариях.
8. **Модерация** — удаление чужих постов и комментариев модераторами и администраторами; назначение ролей администратором.

Для автоматизации в рамках данного проекта выделен **бизнес-процесс «Социальное взаимодействие пользователей»**, объединяющий публикацию контента, реакции, комментарии, подписки, обмен сообщениями и получение уведомлений.

**Основные категории пользователей:**

| Роль | Функции |
|------|---------|
| Участник (participant) | Регистрация, публикации, комментарии, реакции, подписки, сообщения, управление своим профилем |
| Модератор (moderator) | Все функции участника + удаление любых постов и комментариев |
| Администратор (admin) | Все функции модератора + назначение ролей пользователям |

**Бизнес-правила:**

1. Каждый пользователь может иметь только одну учётную запись с уникальным адресом электронной почты.
2. Пользователь может удалять только собственные посты; модератор и администратор — любые посты.
3. На один объект (пост или сообщение) пользователь может поставить не более одной реакции; повторный выбор того же типа снимает реакцию.
4. Комментарий может содержать ответ (parent_id); удаление родительского комментария каскадно удаляет ответы.
5. Подписка асимметрична: пользователь A может подписаться на B без взаимной подписки.
6. Пользователь не может подписаться на самого себя.
7. Личное сообщение имеет отправителя и получателя; при открытии диалога сообщения помечаются прочитанными (read_at).
8. Уведомления отправляются только при включённой соответствующей настройке у получателя.
9. Размер одного изображения в посте не превышает 5 МБ; вложение в чате — 10 МБ.
10. В посте допускается не более 9 изображений.

**Информационные объекты предметной области:**

| Объект | Атрибуты | Назначение |
|--------|----------|------------|
| Пользователь | id, name, email, password, role, avatar, bio, interest_tags, notify_on_* | Учётная запись, профиль, настройки |
| Публикация (Post) | id, user_id, content, images, created_at | Запись в ленте |
| Комментарий | id, post_id, user_id, parent_id, body | Обсуждение публикации |
| Подписка (Follow) | follower_id, following_id | Связь «подписчик — автор» |
| Сообщение | id, sender_id, receiver_id, body, type, attachment_*, read_at | Личная переписка |
| Реакция | id, user_id, reactable_type, reactable_id, kind | Оценка поста или сообщения |
| Уведомление | id, type, data, read_at | Системное оповещение |

**Задачи, решаемые в информационной системе:**

1. *Формирование ленты публикаций* — выборка постов всех пользователей с сортировкой по дате, пагинация (15 записей на страницу).
2. *Создание публикации* — валидация текста и изображений, сохранение файлов в storage/app/public, запись в таблицу posts.
3. *Комментирование* — добавление комментария или ответа, уведомление автора поста.
4. *Постановка реакции* — upsert в таблицу reactions, уведомление автора поста.
5. *Подписка* — проверка уникальности пары (follower_id, following_id), уведомление о новом подписчике.
6. *Отправка сообщения* — сохранение записи, broadcast-событие MessageSent на канал получателя.
7. *Поиск пользователей* — LIKE-запрос по name, email, bio (минимум 2 символа, максимум 40 результатов).

**Входные документы:** регистрационная форма, форма входа, форма создания поста, форма комментария, форма сообщения, форма редактирования профиля.

**Выходные документы (экранные формы):** лента публикаций, страница профиля, список диалогов, окно чата, список уведомлений, результаты поиска.

### 1.2 Проектирование бизнес-процессов предметной области

Для проектирования программного продукта применена методология RUP; построена **диаграмма вариантов использования** (см. приложение А и файл `vkr/grafika/01-use-case.mmd`).

**Акторы:**

- **Гость** — неавторизованный посетитель (просмотр главной страницы, регистрация, вход).
- **Участник** — зарегистрированный пользователь с базовыми правами.
- **Модератор** — пользователь с правом удаления чужого контента.
- **Администратор** — пользователь с правом назначения ролей.

**Основные варианты использования участника:**

- UC-01 «Зарегистрироваться в системе»
- UC-02 «Аутентифицироваться»
- UC-03 «Создать публикацию»
- UC-04 «Прокомментировать публикацию»
- UC-05 «Поставить реакцию»
- UC-06 «Подписаться на пользователя»
- UC-07 «Отправить личное сообщение»
- UC-08 «Просмотреть уведомления»
- UC-09 «Найти пользователя»
- UC-10 «Редактировать профиль»

**Варианты использования модератора** (наследуют участника):

- UC-11 «Удалить чужую публикацию»
- UC-12 «Удалить чужой комментарий»

**Варианты использования администратора:**

- UC-13 «Назначить роль пользователю»

Связи «include»: UC-03, UC-04, UC-05, UC-07 включают UC-02 (требуется аутентификация).

### 1.5 Общие требования к программному продукту

#### Обозначения и указания

Программный продукт **MONO** — веб-приложение класса «социальная платформа», развёртываемое на сервере организации. Обозначение в документации: MONO v1.0.

#### Функциональные возможности

1. Регистрация, аутентификация, восстановление пароля (Laravel Breeze).
2. Лента публикаций с пагинацией.
3. Создание постов с текстом и галереей изображений (lightbox).
4. Реакции четырёх типов на посты и сообщения.
5. Древовидные комментарии.
6. Профили, подписки, поиск пользователей.
7. Личные сообщения: текст, файлы, изображения, голос; read receipts.
8. Realtime-доставка сообщений (Laravel Echo + Pusher/Soketi).
9. Система уведомлений (database + broadcast + browser Notification API).
10. Ролевая модель: участник, модератор, администратор.
11. REST API (Sanctum): feed, posts, users, like.

**Граничные значения:**

| Параметр | Значение |
|----------|----------|
| Максимум изображений в посте | 9 |
| Размер изображения поста | 5 МБ |
| Размер вложения в чате | 10 МБ |
| Длина текста поста | 5000 символов |
| Тегов интересов в профиле | 12 |
| Результатов поиска | 40 |
| Постов на странице ленты | 15 |
| Типы реакций | 4 (heart, fire, clap, eyes) |

#### Надёжность

- Валидация входных данных на сервере (Form Request, правила Laravel).
- Хеширование паролей (bcrypt).
- CSRF-защита форм.
- Каскадное удаление связанных записей (FK constraints).
- Транзакционное сохранение постов с файлами.
- При ошибке загрузки файла — откат записи в БД.
- Восстановление пароля через email-токен.

#### Эффективность

- Время отклика страницы ленты при профиле нагрузки до 100 одновременных пользователей — не более 2 с (локальный сервер OSPanel, PHP 8.1, MySQL 8).
- Доставка сообщения через WebSocket — не более 500 мс при активном соединении.
- Eager loading связей (with) для снижения числа SQL-запросов.

#### Технические и программные средства

| Компонент | Требование |
|-----------|------------|
| Процессор | x64, 2 ядра и выше |
| ОЗУ | 4 ГБ минимум, 8 ГБ рекомендуется |
| Диск | 2 ГБ для приложения + место под uploads |
| ОС сервера | Windows (OSPanel) / Linux |
| PHP | 8.1 и выше |
| MySQL / MariaDB | 8.0 / 10.6+ |
| Node.js | 18+ (сборка frontend) |
| Composer | 2.x |
| Веб-сервер | Apache / Nginx |
| WebSocket (опционально) | Soketi или Pusher Cloud |

**Поставляемые компоненты:** исходный код (PHP, Blade, JS), миграции БД, конфигурационные файлы, собранные assets (npm run build), пояснительная записка.

## 2 ЭКСПЕРИМЕНТАЛЬНЫЙ РАЗДЕЛ

### 2.1 Описание программы

Программа MONO реализована по **модульной архитектуре MVC** фреймворка Laravel. Модульная схема приведена в приложении В и файле `vkr/grafika/03-modules.mmd`.

#### Иерархия модулей

```
MONO (веб-приложение)
├── Модуль аутентификации (Auth)
│   ├── RegisteredUserController — регистрация
│   ├── AuthenticatedSessionController — вход/выход
│   └── Password reset controllers — восстановление пароля
├── Модуль ленты (Feed)
│   ├── FeedController::index — отображение ленты
│   └── PostController — создание/удаление постов
├── Модуль взаимодействия (Engagement)
│   ├── CommentController — комментарии
│   ├── ReactionController — реакции
│   └── LikeController — legacy-лайки, API
├── Модуль профиля и социального графа
│   ├── ProfileController — редактирование своего профиля
│   ├── UserProfileController — просмотр, follow/unfollow, роли
│   └── FriendSearchController — поиск
├── Модуль сообщений (Chat)
│   ├── ChatController — список, диалог, отправка
│   └── MessageSent (Event) — broadcast
├── Модуль уведомлений
│   ├── NotificationController
│   └── NewMessage/Follower/Liked/Commented Notification
└── API-модуль (Sanctum)
    ├── API\FeedController
    ├── API\PostController
    └── API\LikeController
```

#### Описание ключевых модулей

**PostController** — процедуры:
- `store(Request)` — валидация content и images (max 9, 5120 KB), сохранение файлов в `storage/app/public/posts`, создание записи Post, возврат HTML-карточки (AJAX).
- `destroy(Post)` — проверка прав (автор или admin/moderator), удаление файлов из storage, удаление записи.

**ReactionController** — процедуры:
- `storeForPost(Post)` — получение kind, upsert Reaction; при новой реакции — PostLikedNotification; toggle при повторе.
- `storeForMessage(Message)` — аналогично для сообщений.

**ChatController** — процедуры:
- `index()` — список собеседников с последним сообщением и счётчиком непрочитанных.
- `show(User)` — история сообщений, markAsRead для входящих.
- `store(User, Request)` — создание Message, dispatch MessageSent, NewMessageNotification.

**UserProfileController** — процедуры:
- `follow(User)` — attach Follow, NewFollowerNotification.
- `unfollow(User)` — detach.
- `updateRole(User)` — только admin, валидация role ∈ {admin, moderator, participant}.

Клиентская часть (`resources/js/app.js`, `echo-chat.js`): Alpine.js-компоненты, fetch для AJAX, Laravel Echo для подписки на каналы `messages.{id}` и `App.Models.User.{id}`.

### 2.3 Руководство пользователя

#### 2.3.1 Установка и запуск

1. Установить OSPanel (или Apache/Nginx + PHP 8.1 + MySQL).
2. Скопировать проект в каталог домена (например, `D:\OSPanel\domains\mono-net`).
3. Выполнить в терминале:
   ```
   composer install
   npm install
   copy .env.example .env
   php artisan key:generate
   ```
4. Настроить `.env`: APP_URL, DB_*, FILESYSTEM_DISK=public.
5. Выполнить:
   ```
   php artisan migrate
   php artisan storage:link
   npm run build
   ```
6. Для realtime: скопировать `.env.websocket.example`, запустить Soketi, `php artisan config:clear`, пересобрать frontend.
7. Открыть в браузере адрес из APP_URL (например, http://mono-net).

#### 2.3.2 Использование программы

**Регистрация и вход:** на главной странице выбрать «Register» или «Log in», заполнить форму.

**Лента:** после входа открывается `/feed`. В верхней форме ввести текст, при необходимости прикрепить до 9 изображений, нажать «Опубликовать».

**Реакции:** навести на блок реакций под постом, выбрать emoji; повторный клик снимает реакцию.

**Комментарии:** ввести текст в поле под постом; для ответа — «Ответить» у комментария.

**Профиль:** перейти по имени пользователя; кнопка «Подписаться»/«Отписаться».

**Поиск:** пункт меню «Поиск», ввести минимум 2 символа.

**Сообщения:** «Сообщения» → выбрать диалог или начать новый с профиля пользователя. Поддерживаются вложения и голосовая запись (кнопка микрофона).

**Уведомления:** иконка колокольчика; «Прочитать все».

**Профиль (свой):** «Profile» → изменить имя, bio, аватар, теги, настройки уведомлений.
